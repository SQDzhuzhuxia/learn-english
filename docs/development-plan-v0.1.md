# 开发方案与开发计划 v0.1

日期：2026-05-27

## 1. 目标

本项目按接近大厂的软件开发流程推进，但流程必须服务实际交付，不做无意义的形式主义。

目标是把“个人 AI 英语沉浸训练 APP”从想法稳定推进到可每天使用的 Web/PWA 产品，并为未来移动端、桌面端、云同步、离线模型、开源发布留下清晰结构。

第一阶段成功标准：

- 用户每天打开后知道今天该学什么
- 能导入或选择英语材料
- 能进行阅读/听力输入
- 能查词、看 AI 解释、保存词句
- 能进行基础复习
- 能记录学习进度
- 代码结构清晰，可持续迭代

## 2. 开发原则

### 2.1 产品原则

- 输入优先：大量可理解听读输入是产品核心
- 输出辅助：跟读、复述、写作、口语用于验证吸收效果
- AI 降低门槛：AI 负责解释、纠错、生成复习内容、安排学习路径
- 每天能用：任何功能都要服务每日 30 到 60 分钟学习
- 中文母语友好：初期中文为主，中英混合，逐步提高英文比例
- 美国方向：美式英语、美国生活、工作、移民和入籍场景优先

### 2.2 工程原则

- 先跑通闭环，再追求高级功能
- 先做 Web/PWA，再考虑移动端和桌面端
- 先做简单可维护实现，再抽象复杂架构
- 所有 AI 供应商通过统一适配层接入
- 所有核心数据都有清晰模型和迁移记录
- 所有关键流程要能测试、能复现、能回滚
- 代码按未来可开源结构组织

### 2.3 质量原则

- 核心学习流程不能坏
- 用户数据不能丢
- API Key 不能泄露
- AI 输出不能直接污染核心数据，必须经过结构化处理
- UI 必须适配桌面和手机浏览器
- 每个版本都要有明确验收标准

### 4.6 任务拆分

每个功能都按 Epic -> Story -> Task 拆分。

示例：

```text
Epic：材料学习闭环
Story：作为学习者，我希望导入一篇英文文章，并在阅读时查词和保存句子
Task：
- 创建材料数据模型
- 创建材料库页面
- 实现文本导入表单
- 实现阅读学习器
- 实现选词查词
- 实现保存词句
- 编写测试
```

### 4.7 开发

开发要求：

- 使用 Git 分支
- 小步提交
- 每个功能有清晰验收
- 重要逻辑有测试
- UI 修改要浏览器验证
- 不把密钥、录音、本地数据库提交到仓库

### 4.8 Code Review

当前由 Codex 自查，未来可通过 GitHub PR 做 review。

检查清单：

- 功能是否符合需求
- 是否有不必要复杂度
- 是否破坏已有流程
- 错误处理是否充分
- 数据是否可能丢失
- 是否泄露 API Key
- 手机端是否可用
- 是否有测试或手工验证记录

### 4.9 测试

测试分层：

- Unit Test：工具函数、复习算法、文本解析、AI 输出解析
- Integration Test：材料导入、保存词句、生成复习卡
- E2E Test：今日学习流程、阅读学习流程、复习流程
- Manual QA：手机浏览器、桌面浏览器、录音权限、PWA 安装

推荐工具：

- Vitest
- React Testing Library
- Playwright
- ESLint
- TypeScript

### 4.10 发布

发布策略：

- `main` 分支保持可运行
- 每个稳定版本打 tag
- 使用 changelog 记录变化
- 发布前跑完整检查

版本命名：

- `v0.1.0`：本地可运行 MVP 骨架
- `v0.2.0`：输入学习闭环
- `v0.3.0`：复习闭环
- `v0.4.0`：输出练习闭环
- `v0.5.0`：云同步和 PWA

### 4.11 反馈和迭代

每个版本都要回答三个问题：

- 这个版本每天是否真的用得起来？
- 学习者是否更清楚今天该干什么？
- 哪个地方阻碍继续学习？

反馈进入 backlog，再排优先级。

## 5. Git 工作流

### 5.1 分支策略

推荐：

- `main`：稳定分支
- `feature/*`：功能分支
- `fix/*`：修复分支
- `docs/*`：文档分支

## 9. Definition of Ready

一个任务进入开发前，需要满足：

- 功能目标明确
- 用户价值明确
- 页面或交互大致清楚
- 数据影响清楚
- 验收标准清楚
- 依赖项清楚

不满足 Ready 的任务先补需求，不直接开发。

## 10. Definition of Done

一个任务完成，需要满足：

- 功能实现
- TypeScript 无明显类型问题
- lint 通过
- 关键逻辑有测试或明确手工验证
- 桌面和手机基本可用
- 文档或注释在必要处更新
- 没有提交密钥、本地数据、构建产物
- Git 状态清晰

## 12. 安全和隐私

### 12.1 API Key

- 不提交 `.env`
- 不在前端暴露私有 API Key
- 服务端代理 AI 请求
- 设置页只保存必要配置

### 12.2 用户数据

- 学习记录、词句、录音、转写属于个人敏感数据
- 云同步前要明确数据表边界
- 支持后续导出和删除数据
- 录音文件默认可删除

### 12.3 AI 安全

- AI 输出不能默认可信
- 结构化数据要校验
- 失败时给出可理解错误
- 对费用高的请求做限制和提示

## 13. 可观测性和运维

第一阶段本地开发为主，仍保留运维思路。

后续需要：

- 错误日志
- AI 请求耗时和失败率
- Token 消耗统计
- 学习流程完成率
- 页面加载性能
- PWA 安装情况

不要一开始过度接监控平台，但代码结构要方便后续加入。

## 15. 风险清单

### 风险 1：功能太多，第一版做不完

应对：

- 严格按 v0.1 到 v0.5 分阶段
- 第一阶段只跑通每日学习闭环

### 风险 2：AI 成本和速度不稳定

应对：

- 统一 provider adapter
- 支持多供应商切换
- 缓存解释结果
- 控制 prompt 长度

### 风险 3：学习内容太难，用户坚持不下去

应对：

- 材料难度分级
- 初期故事和生活对话优先
- AI 解释用中文降低门槛

### 风险 4：复习系统变成背单词软件

应对：

- 复习单位以句子、短语、真实上下文为主
- 单词必须绑定原句和场景

### 风险 5：多端同步复杂度上升

应对：

- 第一阶段先 Web/PWA
- 数据模型提前设计
- 云同步放到 v0.5

### 风险 6：本地离线模型投入过早

应对：

- 先用云端 API 验证学习效果
- 本地模型和 whisper.cpp 放到后续增强
